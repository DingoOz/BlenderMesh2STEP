# Mesh-to-Parametric Conversion Tool: Plan and Background

## Goal

Investigate whether it is feasible to design and build a tool that converts mesh
models (specifically 3D-printable models authored in Blender) into parametric
representations, and if so, scope a realistic implementation plan.

## The Core Difficulty

What is being described is essentially **reverse engineering**: going from a mesh
(a soup of triangles with no encoded intent) back to parametric features
(sketches, extrudes, fillets, a construction history). In the general case this is
an unsolved problem, for a fundamental reason: the mesh has discarded the
information we want to recover.

A 20mm cylinder and a 64-sided prism that approximates it are *identical* as meshes
if the tessellation matches. The mesh does not know it was "meant to be" a cylinder,
and it does not know the cylinder was created by extruding a circle, revolving a
rectangle, or via a boolean of two other shapes. There are infinitely many
parametric histories that produce the same mesh, and "intent" is exactly what is
missing. Any tool here is fundamentally making *guesses* about intent, and the
quality of the tool is the quality of those guesses.

The problem splits into two very different sub-problems. Conflating them is where
most ambition fails.

### 1. Mesh -> BREP (geometry recovery)

Fit clean analytic surfaces (planes, cylinders, cones, spheres, tori) to regions of
the mesh, then stitch them into a watertight boundary-representation (BREP) solid.
This is "retopology to CAD" and is **tractable**: it is essentially segmentation,
plus surface fitting, plus topology reconstruction.

### 2. BREP -> feature tree (intent recovery)

Take the clean solid and infer the *editable history*: "this is a base extrude of a
sketch, with a fillet on these edges, minus a counterbored hole." This is the
genuinely hard, arguably-unsolved part, and it is an active machine-learning
research area.

Most of the value for a 3D-printing workflow lives in sub-problem 1, and
sub-problem 1 is very doable.

## Why This Use Case Is in the Easy Regime

The hard, money-burning version of this problem is *scan* reverse engineering:
noisy point clouds, missing regions, no clean topology. This is why commercial
tools like Cyborg3D advertise mesh repair and gap-filling as core features, and why
the established players (Geomagic Design X, Siemens NX, Creo, Fusion 360,
PowerShape) lean on kernel-backed healing.

Meshes authored in Blender are *clean*: watertight (or close to it), flat where they
should be flat, with crisp edges and no sensor noise. That removes the entire hardest
layer of the problem. The task is not reverse-engineering reality; it is recovering
structure from geometry generated by simple operations in the first place. That is a
meaningfully more tractable target.

## Existing Tools and Building Blocks

### Geometry kernel (do not write this yourself)

**OpenCASCADE (OCCT)** is the open-source BREP kernel that FreeCAD is built on. It
handles surface fitting, sewing faces into solids, and STEP export. OCCT and CGAL
are the standard libraries for geometry operations in these pipelines. The
architecture is therefore: the tool makes segmentation and fitting decisions, OCCT
does the heavy geometric lifting and produces the BREP/STEP output.

### Reference implementations

- **phossystems/Reverse** (Fusion 360 add-in): reconstructs BREP surfaces from
  selected mesh points, generating geometry within roughly 1e-6mm. The user
  activates a feature type and selects the mesh points belonging to that feature.
  This semi-automatic, human-in-the-loop model is a strong MVP pattern.
- **Professional tools (Solid Edge, etc.)**: the standard workflow is to extract
  faces from identified regions and fit analytical faces on top of a region, rather
  than attempting fully automatic recovery.
- **Commercial mesh-to-CAD (Cyborg3D, RESURF, InStep Studio 3D)**: NURBS/BREP
  reconstruction with STEP/IGES/SAT export, aimed mainly at scan data.

### Mixed modeling (a possible reframing)

Tools like Onshape now treat mesh parts as first-class citizens: you can build new
features (booleans, splits, hole creation) directly on top of a mesh rather than
treating it as an uneditable blob. Sometimes the real need is not full conversion
but slapping a clean parametric feature onto a mesh and performing a boolean. That
is far cheaper than full conversion and may solve the actual problem.

## Proposed Design, in Tiers

Ordered easiest to hardest. Recommendation: stop after Tier 2 unless a specific
need drives you further.

### Tier 1 — Mesh -> clean BREP solid (very achievable)

- Segment the mesh into regions by clustering coplanar / co-cylindrical faces
  (group triangles by normal direction and connectivity; merge adjacent ones whose
  normals agree within a tolerance).
- Classify each region as plane / cylinder / cone / sphere / torus by least-squares
  fitting and selecting the lowest-residual primitive.
- Hand the fitted surfaces to OCCT to trim, sew, and produce a watertight solid,
  then export STEP.
- Result: a model FreeCAD can open as a real solid. Not a parametric history, but
  exact analytic faces that can be measured, booleaned against, and edited
  non-destructively. For 3D printing, this is already most of what people want.

### Tier 2 — Recover 2D sketches + extrude/revolve (achievable with effort)

- Detect the dominant pattern. A set of parallel planar faces connected by faces
  perpendicular to one axis is almost certainly an extrude.
- Project the profile onto a plane, fit lines and arcs to the boundary, and emit
  "sketch + extrude length."
- Rotational symmetry about an axis indicates a revolve.
- This is where genuinely *parametric* output begins (e.g. change the extrude depth,
  edit the sketch dimension). It works well for prismatic, mechanical-ish parts
  common in printing and breaks down on organic shapes.

### Tier 3 — Full feature-tree / intent recovery (research territory; avoid)

- Inferring fillets-as-fillets, hole features, patterns, boolean history, ordering.
- This is the ML-research frontier. The payoff-to-effort ratio falls off a cliff,
  and a printing workflow rarely needs it.

## Recommendation

Build **Tier 1**, with a semi-automatic selection mode like the Reverse add-in, as a
Blender add-on (Python) that shells out to OCCT via **pythonocc-core** or
**CadQuery** (which wraps OCCT with a nicer API and emits STEP directly). Make the
fully-automatic segmentation a convenience the user can correct, not a promise.

This is a weekend-to-a-few-weekends project for a competent developer, produces
immediately useful output, and sidesteps the unsolved part of the problem entirely.

Reframing worth considering: if the real goal is "mesh -> editable solid I can modify
and reprint" rather than "mesh -> parametric," the mixed-modeling approach may be
cheaper and sufficient. You may not need to convert the whole mesh; you may only need
to add a clean parametric feature and boolean it against the mesh.

## Implementation Stack Summary

| Layer                 | Choice                                              |
|-----------------------|-----------------------------------------------------|
| Geometry kernel       | OpenCASCADE (OCCT)                                   |
| Kernel binding        | CadQuery or pythonocc-core (Python); OCCT C++ direct |
| Host / UI             | Blender add-on (Python)                             |
| Output format         | STEP (opens as a true solid in FreeCAD)             |
| Optional geometry ops | CGAL                                                |

## Next Steps

A concrete starting skeleton for the Tier 1 segmentation-and-fitting pipeline can be
written in either C++ (OCCT directly) or Python (CadQuery / pythonocc-core).
